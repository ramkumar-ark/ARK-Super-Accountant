# Feature Landscape

**Domain:** Indian GST/TDS compliance reporting — Tally Prime data analysis for CAs and SME owners
**Researched:** 2026-04-09
**Confidence note:** Web search tools were unavailable during this research session. All findings are from training data (cutoff August 2025) and direct codebase analysis. Indian GST/TDS regulatory knowledge is HIGH confidence (stable statutory rules); specific portal UX conventions are MEDIUM confidence.

## Table Stakes

Features CAs and accountants expect as baseline. Missing any one of these means the product is not usable for professional compliance work.

| Feature | Why Expected | Complexity | Notes |
|---------|--------------|------------|-------|
| GSTIN format validation (15-char regex + state code check) | Every GST transaction requires a valid GSTIN; CA will catch any tool that lets invalid GSTINs through | Low | Already built in GstValidationService. State code (first 2 digits) should be validated against the 37 state/UT codes, not just the regex. |
| GSTR-2B reconciliation: Matched / Unmatched / Excess ITC columns | The GST portal auto-populates 2B; reconciling it against purchase register is the monthly mandatory CA task — before claiming ITC | High | The three-way split (in 2B and in Tally, in 2B but not Tally, in Tally but not 2B) is the minimum output format CAs recognize |
| GSTR-2B recon: GSTIN + invoice number + invoice date + taxable value + IGST/CGST/SGST as match keys | CAs match records on these 5 fields; anything less produces false positives/negatives | High | Invoice number matching must be fuzzy — Tally and the portal often differ in spacing, leading zeros, case. Exact string match will give unacceptably high false-mismatch rates. |
| GSTR-2B recon: ITC eligibility flag (eligible / ineligible / IGST on imports) | Ineligible ITC (blocked credits under Section 17(5)) must be separately identified — car purchases, food, etc. | Medium | The 2B JSON from the portal carries this flag; the reconciliation output must surface it |
| TDS section identification (194C, 194J, 194H, 194I, 192B, 193, 194, 194A, 194B, 194D, 194DA, 194EE, 194F, 194G, 194LA, 194LB) | 26Q/27Q returns are filed section-wise; a CA who can't see section-wise totals cannot prepare the return | High | Section must be derived from ledger/nature-of-payment in Tally; mapping table is required |
| TDS deductee-wise breakdown (PAN, name, amount paid, TDS deducted, TDS rate, section) | The 26Q annexure (Annex I) is exactly this format; PAN is the deductee identifier | High | PAN validation (10-char alphanumeric with 4th char = initial of surname type, 5th = initial of first name) must be enforced — missing/invalid PAN triggers higher TDS rate (20%) |
| TDS liability summary (total amount paid per section, total TDS deducted, TDS payable, TDS already paid via challan) | CAs need to reconcile what was deducted vs what was deposited | Medium | Challan data typically lives outside Tally; v1 can show liability only, flagging challan reconciliation as manual |
| HSN/SAC code presence on taxable invoices (above ₹5 lakh threshold) | GST law mandates HSN summary in GSTR-1; CAs get audit queries if HSN is missing | Medium | HSN validation: 4-digit minimum for businesses with turnover 1.5–5 Cr, 6-digit for above 5 Cr |
| Place of Supply determination and inter/intra-state classification | Wrong classification = wrong tax type (IGST vs CGST+SGST); already partially built | Medium | Already exists in GstValidationService (checkIncorrectGstType). Needs to handle special cases: SEZ, overseas supplies, export |
| Export invoice zero-rating and LUT bond linkage | Exporters claim zero-rated status under LUT or with IGST paid; the system must not flag zero-GST exports as errors | Medium | Voucher type "Export" or place of supply "Outside India" — must be excluded from standard GST-not-charged error |
| RCM (Reverse Charge Mechanism) tracking | RCM liability is self-assessed by the recipient; it is a mandatory disclosure in GSTR-3B. Missing RCM entries are a common audit finding | Medium | Already partially built (checkRcmMixedWithTaxable). Needs to also flag: RCM applicable but no self-invoice raised |
| E-invoice IRN presence for B2B invoices above ₹5 Cr threshold | E-invoicing mandatory for large taxpayers; IRN (Invoice Reference Number) must be on the voucher | Medium | IRN is a 64-char hash generated by the IRP. The system should flag B2B invoices above the threshold that lack an IRN field |
| Credit/Debit note linkage to original invoice | A credit note must reference the original invoice number and date; unlinked notes are a common GST filing error | Medium | Cross-reference check within the Tally data |
| Multi-client switching without re-login (for CA mode) | CAs manage 10–50 clients; re-logging for each client is a session-killer. Competitors (Zoho Books, ClearTax) all support this | Medium | Organization model exists; CA ↔ multi-client relationship needs to be modeled |
| Downloadable/exportable reconciliation report (Excel or CSV) | CAs share reports with clients and file them; a web-only view that cannot be exported is not usable in a CA workflow | Low | CSV export already pattern-exists in the masters upload pipeline |
| Role-based access: read-only Auditor/CA role | External CA reviewing client books must not be able to modify data; a missing read-only role means a CA won't give the tool to clients | Low | ERole needs ROLE_AUDITOR added; @PreAuthorize guards needed (currently absent — see CONCERNS.md) |

## Differentiators

Features that give Super Accountant a competitive edge over ClearTax, Zoho Books GST module, and TallyPrime's own reports. Not expected by users at first, but become reasons to stay.

| Feature | Value Proposition | Complexity | Notes |
|---------|-------------------|------------|-------|
| Fuzzy invoice number matching in GSTR-2B recon | Most tools do exact string match; Tally users enter "INV-001", portal shows "INV001" — 30–40% false mismatches with exact match | High | Levenshtein distance or token normalization (strip hyphens, leading zeros, spaces, case-fold). Configurable tolerance threshold per client. |
| AI invoice extraction with accounting entry generation | Eliminates data entry for purchase invoices; no competitor does this end-to-end for Tally import | High | Must handle: photo of paper invoice (mixed quality), scanned PDF, digital PDF. Output: structured JSON that maps to Tally voucher format. |
| AI audit trail per extracted invoice | CAs need to know why the AI chose a specific ledger or HSN code; without this they cannot review AI output | Medium | Log: extracted fields, confidence scores, ledger mapping rationale, any ambiguities flagged. This is already in PROJECT.md scope. |
| TDS section auto-detection from payment ledger name | Tally stores payments with ledger names like "Subcontractor Charges" — mapping to 194C requires a rules engine or ML; most CA tools require manual section entry | High | Rule table: keyword → section mapping. Fallback: flag as "section ambiguous — review required." This prevents wrong-section filing, which attracts penalties. |
| GSTR-2B ITC claim vs ledger amount variance alert | Beyond matched/unmatched: flag cases where the invoice is matched but the ITC claimed differs from what the 2B says (partial payments, debit notes, etc.) | Medium | This is the "soft mismatch" — invoice exists on both sides but amounts differ. ClearTax shows this; it is the most actionable finding type. |
| Exempt supply identification and GSTR-1 table routing | Exempt, nil-rated, and non-GST supplies go to different GSTR-1 tables (Table 8 for exempt); incorrect routing is a common filing error | Medium | Voucher-level classification: taxable / exempt / nil-rated / non-GST / zero-rated export |
| Tally re-import JSON: round-trip corrected entries | After AI extraction or reconciliation, push corrected vouchers back to Tally without manual re-entry | High | Tally Prime JSON import format must be exactly followed; any schema deviation causes silent import failure in Tally. Needs a test harness with sample Tally imports. |
| Composition scheme dealer detection and GSTR-4 flag | Composition dealers file quarterly GSTR-4, not monthly GSTR-1/3B; a tool that treats them as regular taxpayers produces wrong output | Low | GST registration type field in Tally ("Composition") triggers a different validation and report set |
| Deductor-vs-collected TDS reconciliation (Form 26AS context) | A CA's client will often get a mismatch notice from IT dept because the vendor didn't deposit TDS; flagging this gap proactively is high value | High | Requires comparing deductee PAN + TDS amount against Form 26AS data. Form 26AS is downloadable from TRACES as text; parsing it is a separate feature. V1: flag the need, don't build the parser. |
| Manager approval workflow for AI-generated entries | AI extracts → Data Entry Operator reviews → Manager approves → pushed to Tally. This workflow matches how CA firms actually work. | High | New ROLE_MANAGER needs approve/reject action on AI-extracted vouchers before they enter the re-import JSON |

## Anti-Features

Things that appear useful but create problems — either regulatory overreach, hidden complexity, or scope creep that delays the core value proposition.

| Anti-Feature | Why Avoid | What to Do Instead |
|--------------|-----------|-------------------|
| GSTR-1 / GSTR-3B auto-filing (direct portal submission) | Requires GSP (GST Suvidha Provider) license from GSTN, which is a regulatory body approval process taking months and involving annual audits. Also introduces liability if a wrong return is filed automatically. | Generate the report and the JSON/Excel in the format the CA can upload manually to the portal. Mark direct filing as v3+ after GSP licensing. |
| TDS challan generation and TRACES integration | TRACES API access requires TAN-level credentials from the deductor and TDSCPC/TRACES registration. Wrong challan = penalty + interest. Manual challan filing takes 2 minutes; an incorrect auto-challan causes weeks of rectification. | Show TDS liability summary with "challan required by [date]" alert. Let the CA file the challan manually via TRACES. |
| 26Q / 27Q FVU file generation (for TIN-NSDL submission) | FVU (File Validation Utility) is a Java-based NSDL tool with version-specific validation rules that change with each budget. Generating an FVU-compatible file that passes their validation is a moving target. The penalties for wrong returns are significant. | Show the 26Q-format data (Annexure I with all required fields) in a report. The CA uses TDS software (KDK, Winman, Saral TDS) to prepare the actual FVU file. |
| Real-time GST portal API polling (live return status) | GSTN API access requires NIC-level API partner agreement, rate limits are strict (10 req/min), and portal availability is notoriously poor during filing deadlines. Building around GSTN API SLAs will cause reliability incidents at exactly the wrong time. | Use portal JSON exports (GSTR-2B download is a single JSON file). The CA downloads it, uploads to Super Accountant. No portal API needed. |
| Full P&L / Balance Sheet / Trial Balance generation | This is what Tally already does well. Competing with Tally's core reports means building a general ledger — massive scope, and the CA already has Tally open. | Focus on compliance-specific reports (GST, TDS) that Tally's built-in reports do not generate in CA-friendly formats. |
| Automatic e-invoice generation and IRN fetching | E-invoicing requires integration with the IRP (Invoice Registration Portal) via API, including digital signature of the invoice JSON (JWS). This is a product in itself (ClearTax charges separately for this). | Validate that IRN is present on imported vouchers. Flag missing IRN. Do not generate IRN — that is the source system's (Tally's) responsibility. |
| Customer-facing GST reconciliation portal (B2B buyer access) | Giving buyers access to view their purchase reconciliation from the seller's books crosses organizational data boundaries and creates a privacy/data-sharing surface that requires explicit consent flows and potentially a separate GSTIN-based portal. | Stick to single-entity reconciliation: one GSTIN's books against its GSTR-2B. Multi-entity linking is v4+. |
| AI-generated tax advice ("you should claim this ITC") | Providing tax advice without a CA license is legally problematic in India (CA Act 1949). The AI can surface data; it cannot recommend tax positions. | Phrase all AI output as "extracted data" and "calculated figures." Include a disclaimer. Flag anomalies as "review required" not "this is wrong." |
| Mobile-first / native mobile app | CAs do reconciliation work at desks with multi-screen setups; the data volumes (hundreds of invoices) and table-heavy UI are not mobile-appropriate for v1. Building mobile doubles the surface area. | Responsive web is sufficient. Mobile-optimized upload (photo capture for AI invoice processing) can be a later add-on. |
| Historical GSTR-2A data migration and multi-year recon | GSTR-2A (supplier-filed data) predates GSTR-2B; reconciling both requires understanding the transition rules (2B replaced 2A for ITC from Oct 2021). Multi-year recon is a data archaeology project. | GSTR-2B only, current period + up to 3 months rolling. Multi-year is v2 after the monthly workflow is proven. |

## GSTR-2B Reconciliation Report: Required Content

Confidence: HIGH (statutory requirement, stable since Oct 2021)

The minimum viable GSTR-2B reconciliation report that a CA will accept contains:

**Header**
- GSTIN of the taxpayer
- Return period (month/year)
- Date of GSTR-2B generation (from portal JSON)
- Date of reconciliation run

**Per-invoice rows — required columns**
- Supplier GSTIN
- Supplier name (from Tally ledger, not portal)
- Invoice number (from portal / from Tally — shown separately to enable diff inspection)
- Invoice date
- Invoice value (total including GST)
- Taxable value
- IGST / CGST / SGST (portal side)
- IGST / CGST / SGST (Tally side)
- ITC eligible amount (portal carries this — eligible vs blocked)
- Place of supply
- Reverse charge flag
- Match status: MATCHED / AMOUNT_MISMATCH / IN_2B_NOT_IN_TALLY / IN_TALLY_NOT_IN_2B
- Mismatch reason (when applicable)

**Summary section**
- Total ITC available as per GSTR-2B
- Total ITC booked in Tally (purchases)
- Difference (potential excess or short ITC claim)
- Count of matched invoices
- Count of mismatched invoices (with total value)
- Count of invoices only in 2B (supplier filed, buyer missed)
- Count of invoices only in Tally (ITC may be at risk — supplier not filed)

**What CAs specifically look for that most tools miss**
- The "in Tally, not in 2B" bucket is high priority: it means the CA booked a purchase but the supplier has not filed GSTR-1, so this ITC cannot be claimed yet — this is the actionable finding
- Invoice number fuzzy match tolerance setting (so the CA can tune it per client's Tally data quality)
- Net ITC position after reconciliation (how much can be safely claimed this month)

## TDS Report: Mandatory Fields

Confidence: HIGH (Income Tax Act + 26Q form structure, stable)

**Deductee-wise report (maps to 26Q Annexure I)**
- Deductee PAN (10-character, validated)
- Deductee name (as per PAN records — must match exactly for TRACES)
- Deductee type: company / non-company (affects TDS rate applicability)
- Section of the Act (194C / 194J / 194H / 194I / 192B / 194A etc.)
- Nature of payment (description matching the section)
- Amount paid / credited during the period
- Date of payment / credit (whichever is earlier — "deduction trigger" rule)
- TDS rate applied (%)
- TDS amount deducted
- Surcharge (if applicable — for payments above threshold)
- Education cess (if applicable)
- Total tax deducted
- Date of deduction
- Certificate issued (Form 16A) — flag for whether 16A has been generated

**Section-wise summary**
- Section code
- Total number of deductees in this section
- Total amount paid under this section (period)
- Total TDS deducted (period)
- TDS payable (after adjusting for any excess from prior period)

**Special flags that distinguish professional-grade TDS reports**
- PAN not available flag: triggers 20% TDS rate under Section 206AA; must be highlighted as an exception requiring resolution
- Lower deduction certificate flag (Section 197): some vendors have a certificate for reduced TDS rate; the system must store and apply this
- TDS threshold check: Section 194C threshold is ₹30,000 per payment / ₹1,00,000 cumulative with same deductee per year; below threshold = no TDS; the report must show cumulative position
- Nature of payment disambiguation: a payment ledger named "Professional Charges" could be 194J(a) (fees for technical services) or 194J(b) (fees for professional services) — different rates apply (2% vs 10% post-amendment); flagging this ambiguity is essential

## AI Invoice Extractor: What CAs Actually Need

Confidence: MEDIUM (from industry knowledge; no direct user research available)

**What CAs need extracted (minimum)**
- Supplier name (as printed on invoice, not cleaned)
- Supplier GSTIN (15-char, validated on extraction)
- Invoice number
- Invoice date
- Line items: description, HSN/SAC code, quantity, rate, amount
- Taxable value per line
- GST rate per line (5% / 12% / 18% / 28%)
- IGST / CGST / SGST amounts (line level and total)
- Total invoice value
- Place of supply (state, determines inter/intra-state)
- Reverse charge flag ("Y" or "N" — printed on GST invoices)
- E-invoice IRN (if present — a QR code on the invoice)
- E-invoice QR code scan (for verification against IRN)

**What CAs need from the accounting entry generation**
- Ledger mapping suggestion: supplier name → party ledger, each line item → expense/purchase ledger, GST amounts → Input CGST/SGST/IGST ledger
- Confidence score per mapping (so the CA can review low-confidence mappings before import)
- Ability to override the AI mapping before approving for re-import
- The output must be the Tally JSON voucher format — not a proprietary format that requires another conversion step

**What makes CAs reject AI invoice tools**
- Hallucinated amounts (AI "completes" a partially visible number incorrectly): requires confidence threshold — if amount extraction confidence is below 85%, flag for human review
- Wrong HSN code: HSN codes are 4-8 digits; HSN classification affects GST rate; a wrong HSN code in Tally flows into GSTR-1 filing — this is a filing error with potential penalty exposure
- Missing line items: a multi-line invoice where the AI extracts only the first line and the total — the CA cannot split the entry correctly by expense category
- No audit trail: if the CA cannot see what the AI extracted from the raw image vs what it computed, they will not trust or use the tool

**Practical constraints from the Indian invoice reality**
- Paper quality: many vendor invoices in India are handwritten or printed on dot-matrix printers — low contrast, smeared ink
- Language mixing: GST invoice descriptions often mix English and regional language (Hindi, Tamil, Marathi etc.) for item names
- Format diversity: no standard invoice template; each vendor has their own layout
- Pre-GST invoices: some vendors issue non-GST bills (URD purchases); the extractor must handle invoices with no GSTIN and no GST amounts without error

## MVP Recommendation

Given that CAs will evaluate the tool primarily on reconciliation accuracy and report readiness:

**Phase 1 must have (without these, CAs will not evaluate further)**
1. GSTR-2B reconciliation with the three-bucket output (matched / in-2B-not-Tally / in-Tally-not-2B) using fuzzy invoice number matching
2. TDS deductee-wise report with section identification from Tally ledger names (even if the mapping table starts simple)
3. Multi-client switching for CA role
4. CSV/Excel export of reconciliation report

**Phase 2 (can be promised, not required for first CA pilot)**
5. AI invoice extraction with accounting entry generation
6. Manager approval workflow
7. Tally re-import JSON

**Defer deliberately**
- Form 26AS reconciliation: high value, high complexity, needs TRACES data source — v2
- IRN validation (e-invoice QR scan): medium value, requires IRP API or QR decode library — v2
- TDS threshold tracking (cumulative 194C check): important but requires carrying state across periods — v2
- Composition dealer GSTR-4 workflow: niche within the user base — v3
