```tsx
type Role = 'ROLE_OWNER' | 'ROLE_ACCOUNTANT' | 'ROLE_OPERATOR' | 'ROLE_AUDITOR_CA'

interface RoleBadgeProps {
  role: string | undefined
}

const ROLE_CONFIG: Record<Role, { label: string; bg: string; fg: string }> = {
  ROLE_OWNER:      { label: 'Owner',      bg: 'var(--color-warning-bg)', fg: 'var(--color-warning)' },
  ROLE_ACCOUNTANT: { label: 'Accountant', bg: 'var(--color-success-bg)', fg: 'var(--color-success)' },
  ROLE_OPERATOR:   { label: 'Operator',   bg: 'var(--color-primary-light)', fg: 'var(--color-primary)' },
  ROLE_AUDITOR_CA: { label: 'CA Auditor', bg: 'var(--color-info-bg)',    fg: 'var(--color-info)' },
}

export function RoleBadge({ role }: RoleBadgeProps) {
  if (!role || !(role in ROLE_CONFIG)) return null
  const cfg = ROLE_CONFIG[role as Role]
  return (
    <span
      role="status"
      aria-label={`Your role: ${cfg.label}`}
      className="inline-flex items-center px-2 py-0.5 rounded-[var(--radius-sm)] text-xs font-medium uppercase tracking-wide"
      style={{ backgroundColor: cfg.bg, color: cfg.fg }}
    >
      {cfg.label}
    </span>
  )
}
```

## Threats

| Threat | STRIDE | Severity | Mitigation |
|--------|--------|----------|------------|
| T-02-12: Client-side role enumeration — DOM inspection reveals which nav items/actions the user does NOT have access to | Information Disclosure | Low | UI-SPEC mandates hidden items are omitted from the DOM entirely (not just `opacity-0`). Phase 2 introduces no sidebar items yet; rule recorded for Phase 3+. |
| T-02-13: Client trusts role from Zustand to gate writes — attacker edits localStorage, grants themselves `ROLE_OWNER` in DOM | Elevation of Privilege | Low (client-side is UX only) | All authoritative role checks are on the backend via `@PreAuthorize` (Plan 02). Client role is UX only. Attempting writes with escalated local role still returns 403. |
| T-02-14: `switchOrganization` receives an untrusted `role` string from the API response and writes it to Zustand; a compromised API or MITM could inject an arbitrary string | Tampering | Low | API is HTTPS (TLS pinning is out of scope for Phase 2). Even a tampered role string only affects UX; all backend authorization re-resolves role from DB per request. RoleBadge renders `null` for unknown roles (implicit validation). |
| T-02-15: Old JWT still in `localStorage` after org switch — replayable if not overwritten | Session fixation | Low | `switchOrganization` writes the new `token` into Zustand (persisted via `persist` middleware) replacing the old value atomically. No second token lingers. |
| T-02-16: Signup default role `operator` allows someone to self-provision as an Operator without an invite — granting upload/resolve capability | Elevation of Privilege | Medium (accept for now) | Per research Open Question #2, signup without invite flow is considered acceptable for v1; stakeholders can tighten to invite-only via a later phase. Backend still enforces role via `@PreAuthorize`; there is no higher privilege (OWNER/ACCOUNTANT create orgs but self-signup OPERATOR has no org yet → cannot perform any data entry until invited). |

## ASVS L1 Notes

- **V4.1.1 Authorization at trusted service layer** — Client role gates are cosmetic; backend `@PreAuthorize` is the enforcement boundary. RoleBadge rendering `null` for unknown roles prevents garbage UI without implying security.
- **V3.5.2 Token invalidation on context switch** — Org switch replaces the JWT atomically via `switchOrganization`; old token removed from Zustand state.
- **V2.10.1 Privilege escalation prevention** — Client cannot grant itself roles; any attempt to mutate Zustand state is defeated by server-side guards.
